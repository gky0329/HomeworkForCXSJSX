## 1. 项目定位

**目标**：开发一个基于 QT/PySide6 的 C++ 教学工具，借鉴 Learn Git Branching (LGB) 的「抽象数据结构可视化 + 目标驱动谜题」范式。

**首选知识点**：C++ 类与对象（OOP 基础），含 class 定义、成员变量/函数、访问控制、对象实例化。课程已覆盖指针与内存管理，本项目切入点后移。

**核心交互模型**：左侧代码面板（填空式步进执行），右侧 OOP 状态画布（实时可视化类定义、对象实例、成员关系）。

**技术栈**：Python 3.10+ / PySide6 >= 6.6 / QGraphicsView / pytest

## 2. 现有 Demo 问题诊断

对当前 `demo/`（~1860 行 Python，31 文件）的逐文件审查结果：

### 2.1 架构层面（3 项高风险）

| 问题 | 位置 | 影响 |
|---|---|---|
| **God Object**：`MainWindow` 创一切、连一切、管一切 | `app/main_window.py:1-137` | 新增功能需改 5 处，无法多人并行开发 |
| **信号蛛网**：所有页面间导航通过 MainWindow 内硬编码的 lambda 连接 | `main_window.py:56-128` | 难以追踪、无法校验跳转合法性 |
| **状态/逻辑混血**：`AppSession` 同时持有导航状态和答题状态 | `app/models/session.py:55-96` | 测试困难，状态污染风险高 |

### 2.2 代码质量（数据驱动，按严重程度排序）

| 类别 | 问题数 | 典型示例 |
|---|---|---|
| **硬编码值** | 60+ | 窗口尺寸 `(1400,820)`、音量 `0.28`、heart 数 `5`、创意模式 `999`、字体大小、间距全部内嵌在代码中 |
| **零错误处理** | 10+ | `StorageService` 读写 JSON 无 try-except、`LessonService.load_level()` 文件不存在直接 raise |

| **缺失功能** | 7 项 | 配置管理、i18n 国际化、设置页、关卡索引系统、存档管理、打包分发、CI |

## 3. LGB 范式迁移方案

### 3.1 LGB 的核心优势拆解

| LGB 机制 | Git 领域映射 | C++ 教学映射 |
|---|---|---|
| **隐性状态显性化** | 用节点图展示 `.git` 内部 DAG | 用 Class/Instance 双区画布展示 OOP 结构 |
| **隔离沙盒** | 纯前端，零环境配置 | 内嵌伪执行环境，不依赖系统编译器 |
| **目标驱动谜题** | 「用最少的命令达成目标形状」 | 「填空/调整语句使 OOP 状态达到预期」 |
| **步进 + 撤销** | 随时 reset / undo | Step Forward / Step Back / Reset |

### 3.2 C++ 特化的可视化映射

```
Git DAG 模型                     C++ OOP 模型
─────────────                    ─────────────
Commit 节点          ──映射──→   类定义（ClassDef，含成员/方法签名）
Branch 引用          ──映射──→   对象实例（ObjectInstance，堆上/栈上）
HEAD 指向            ──映射──→   当前作用域／最近操作的实例高亮
merge 操作           ──映射──→   继承链（父类→子类成员合并）
有向边               ──映射──→   成员访问关系（对象→成员变量/函数调用）
orphan commit        ──映射──→   未初始化的对象（无构造调用）
Detached HEAD        ──映射──→   悬空引用（对象已销毁仍持有引用）
```

### 3.3 知识点学习路径

```
Phase 1（本 MVP）：类与对象基础
  Unit 1: class 定义与成员变量（声明、访问控制 public/private）
  Unit 2: 对象实例化与成员赋值（栈上对象、. 运算符）
  Unit 3: 成员函数（声明与定义、const 成员函数）

Phase 2（后续）：
  Unit 4: 构造函数与析构函数
  Unit 5: 继承（单继承、访问控制变化）
  Unit 6: 多态（虚函数、纯虚函数/抽象类）

Phase 3（远期）：
  STL 容器内部机制（vector/list/map 的类模板设计）
  指针与内存管理进阶（RAII、智能指针）
```

⭐ **本报告以下所有设计均以 Phase 1 为范围。**

### 4.2 关键架构决策

| 决策 | 选择 | 理由 |
|---|---|---|
| **页面通信方式** | NavigationManager 状态机 | 替代现有的 lambda 蛛网，跳转合法性可校验 |

## 5. 核心引擎设计 ⭐

### 5.1 OopModel

引擎的核心数据结构，保存当前沙盒的完整运行时状态：

```python
@dataclass
class MemberDef:
    name: str            # 成员名
    type: str            # 类型 (int/float/string/...)
    access: str          # public | private

@dataclass  
class ClassDef:
    name: str            # 类名
    members: list[MemberDef]

@dataclass
class ObjectInstance:
    cls_name: str        # 所属类名
    var_name: str        # 变量名
    values: dict[str, Any]  # 成员名 → 当前值

class OopModel:
    classes: dict[str, ClassDef]
    instances: dict[str, ObjectInstance]
    current_scope: str      # 当前所在类（定义时）或 "main"

    def get_class(self, name: str) -> ClassDef: ...
    def get_instance(self, name: str) -> ObjectInstance: ...
    def snapshot(self) -> "OopModel": ...   # deep copy, 用于 undo
```

### 5.2 StepExecutor（MVP 支持的语句）

MVP 阶段支持 7 种语句。解析策略采用**模式匹配（正则）而非完整 C++ 语法解析**——每个语句类型对应一个独立的正则模式，匹配后直接执行语义操作。这种设计的前提约束：

- **不处理组合表达式**：`a = b = c;` 等不被支持
- **不处理嵌套结构**：不支持函数调用、模板等
- **空格处理规范化**：所有模式使用 `\s*` 容忍变量周围的空白

| ID | 语句类型 | 示例 | 操作 |
|---|---|---|---|
| 1 | 类定义 | `class Player { ... };` | 创建 ClassDef，解析成员列表 |
| 2 | 成员变量声明（在类体内） | `int health;` | 向当前 ClassDef 添加 MemberDef |
| 3 | 访问说明符 | `public:` / `private:` | 切换当前类的默认访问控制 |
| 4 | 对象实例化 | `Player p;` | 创建 ObjectInstance，所有成员初始化为默认值 |
| 5 | 成员赋值 | `p.health = 20;` | 修改 ObjectInstance.values[member]，触发 Canvas 更新 |
| 6 | 对象拷贝声明 | `Player q = p;` | 创建新实例并复制所有成员值 |
| 7 | 作用域进出 | `int main() {` / `}` | 创建/退出主作用域，作用域退出时检查未使用对象 |

### 5.3 StateDiff

引擎执行一行代码后输出 StateDiff，Canvas 据此播放动画：

```python
@dataclass
class ClassError:
    kind: str               # "redefinition" | "missing_class" | "access_violation"
    description: str        # 用户可读描述
    class_name: str | None
    line: int

@dataclass
class Annotation:
    kind: str               # "info" | "tip" | "warning"
    text: str
    target: str

@dataclass
class StateDiff:
    added_classes: list[ClassDef]
    added_instances: list[ObjectInstance]
    updated_instances: list[tuple[str, str, Any]]  # (var_name, member, new_value)
    removed_instances: list[str]
    errors: list[ClassError]
    annotations: list[Annotation]
    source_line: int
    source_text: str
```

⭐ **设计要点**：Canvas 只消费 StateDiff，不直接读 OopModel。引擎可脱离 UI 独立单元测试。

### 5.4 错误检测规则

| 检测项 | 触发条件 | 视觉表现 |
|---|---|---|
| **重复定义类** | 同一类名被 class 定义两次 | 类名闪烁红色 + 错误文本 |
| **类未定义** | 实例化尚未定义的类 | 实例创建失败，红色提示"类未找到" |
| **访问越权** | 从外部访问 private 成员 | 成员名红色 + 警告图标 |
| **类型不匹配** | 赋值的值类型与成员声明类型不匹配 | 值显示黄色 + 警告 |

### 6.3 动画编排

| 成员赋值 | 目标成员值闪烁高亮（2 次），值过渡动画 | 200ms |
| 错误触发 | 目标区域红色高亮，闪烁 3 次 | 1000ms |

⭐ **实现方式**：`QPropertyAnimation` 对 `QGraphicsItem` 的 `pos`、`scale`、`opacity` 属性做过渡。

> ⚠️ **动画竞态防护**：`CanvasAnimator` 内部维护 `_animating` 标志位和动画事件队列。当动画播放期间（200-1000ms），UI 层的 Step 按钮被禁用（`setEnabled(False)`），防止用户快速连击导致 Qt 动画引擎坐标计算错乱。动画结束后通过 `finished` 信号解锁按钮并检测队列中是否还有待播放的 diff。

## 7. 关卡系统设计

### 7.1 关卡 JSON 结构

```json
{
  "id": "oop_001",
  "title": "定义你的第一个类",
  "unit": 1,
  "concept": "class 声明与成员变量",
  "order": 1,
  "prerequisites": [],
  "initial_code": [
    "int main() {",
    "    __BLANK__ Player {",
    "        __BLANK__:",
    "            int health;",
    "    };",
    "    Player p;",
    "    p.__BLANK__ = 20;",
    "}"
  ],
  "blanks": [
    {
      "line": 1,
      "type": "text_input",
      "placeholder": "__BLANK__",
      "width_chars": 8,
      "validator": "cpp_keyword"
    },
    {
      "line": 2,
      "type": "text_input",
      "placeholder": "__BLANK__",
      "width_chars": 10,
      "validator": "cpp_access"
    },
    {
      "line": 5,
      "type": "text_input",
      "placeholder": "__BLANK__",
      "width_chars": 10,
      "validator": "identifier"
    }
  ],
  "goal": {
    "checks": [
      { "type": "class_exists", "name": "Player" },
      { "type": "instance_exists", "var_name": "p", "class_name": "Player" },
      { "type": "member_value_equals", "var_name": "p", "member": "health", "value": 20 },
      { "type": "no_errors" }
    ]
  },
  "hints": [
    "提示：声明类的关键字占 5 个字符",
    "提示：让类外能访问 health"
  ],
  "success_message": "很好！你定义了第一个类，并创建了一个 Player 对象，设置 health 为 20。"
}
```

### 7.2 Goal Checker 规则（MVP 阶段支持）

| 检查类型 | 语义 | 参数 |
|---|---|---|
| `class_exists` | 存在指定类定义 | name |
| `class_has_member` | 类包含指定成员 | class_name, member_name, type |
| `instance_exists` | 存在指定对象实例 | var_name, class_name |
| `member_value_equals` | 实例某成员值为指定值 | var_name, member, value |
| `instance_count` | 某类的实例个数 | class_name, count |
| `no_errors` | 无任何 ClassError | 无 |

### 7.3 关卡间依赖

通过 `prerequisites` 字段建立 DAG，关卡选择页可按拓扑序展示。Phase 1 只有一关，此字段预留。
