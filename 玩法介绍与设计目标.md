# 玩法介绍与设计目标

> 版本 v1.0 | 2026-05-14 | 供组内讨论

---

## 一、当前玩法（v2.1 现状）

### 1.1 用户流

```
主菜单 → 单人游戏 → 关卡选择 → NPC 对话 → 选择题 → 填空代码 → Canvas 渲染 → 结果
```

### 1.2 各环节详解

#### NPC 对话（课前阅读）

- 村民和铁傀儡用文字「讲课」。标题 + 几句对话 + 「开始挑战」按钮。
- 用户可以跳过，3 秒读完。

#### 选择题

- 单选（A/B/C/D），逐题弹出，选对显示 ✅，选错显示 ❌ + 正确答案。
- 答完 3 题后自动过渡到代码阶段。
- 没有计分、没有生命值、没有奖励——选了就过了。

#### 填空代码

- 预填代码行中挖去关键 token（如 `class`、`public`、`health`）替换为 `___`。
- 一次只激活一个空白输入框（金色边框），用户键入后点「下一步」前进到下一个空白。
- 所有空白填满后按钮变为「⛏ 执行」，点击运行。

#### Canvas 渲染

- 右侧 QGraphicsView，左半区绿色 `ClassItem` 卡片（类名 + 🔓/🔒 成员列表），右半区蓝色 `InstanceItem` 卡片（变量名 + 值表格）。
- Engine 解析完整代码 → 产出 StateDiff → Canvas 一次性静态渲染所有卡片。
- 没有动画（飞入/闪烁/渐隐已实现但未接入），没有交互。

#### 结果

- 通关：✅ + success_message + 「下一关 ▶」+ 「🔄 重试」
- 失败：❌ + 错误原因 + failure_guidance + 「🔄 重试」
- 最后一关：🎉 全部通关！

### 1.3 当前使用感受

| 维度 | 评价 |
|---|---|
| 学习效果 | 能教语法（class/对象/成员赋值），但教不了设计思维 |
| 交互密度 | 选择题 + 填空题 = 两次考试，中间零反馈 |
| 游戏感 | 几乎为零。MC 主题仅体现在按钮贴图和全景背景 |
| 紧张感 | 无。答错只显示 ❌，没有后果，没有激励再来一次 |
| 视觉反馈 | Canvas 是静态卡片，不交互，只在最后出现 |

## 二、设计目标（讨论稿）

### 2.1 核心理念

> **让用户感觉自己在「建造一个 MC 世界」，而不是在「做一套 C++ 试卷」。**

LearnGitBranching 成功的核心：用户敲真实命令，一棵 commit 树在面前生长。每个命令都在改变可视状态，出错时树形会走偏，做对了立刻有成就感。

我们的映射：
- LGB 的 commit 树 → **MC 世界里活生生的实体**
- LGB 的 git 命令 → **C++ 类定义和对象操作代码**
- LGB 的目标驱动关卡 → **「让世界出现一只苦力怕」**

### 2.2 五个改进方向（供讨论）

#### 方向 A：即时沙盒（推荐优先级：最高）

砍掉选择题阶段的「考试感」。NPC 说一句话 → 画布上立刻出现需要你完成的一行代码 → 你敲键盘的每一个字符，Canvas 都在实时变化。

例：
- NPC：「我们定义一个 Creeper 类吧。」
- Canvas 出现一行 `___ Creeper {`，光标在 blank 上闪烁。
- 你敲 `c` → Canvas 上出现半个绿色卡片轮廓。
- 你敲 `lass` → 完整的 class Creeper 绿卡片出现在 Class 区。
- 不需要「确认」「执行」按钮，打字即反馈。

#### 方向 B：生存机制

复用旧 demo 的心形生命值、难度模式、连击系统。
- 填空题答错 → 扣半颗心 → 屏幕边缘红闪
- 心耗光 → 挑战失败 → 「你被苦力怕炸飞了！」
- 连击（连续 3 题正确）→ 经验值音效 + 生命恢复

#### 方向 C：交互式 Canvas

Canvas 不只是展示，而是可交互的调试工具。
- 点击实例卡片 → 弹出属性面板（health 当前值、所属类、可修改）
- 拖拽实例 → 比较不同实例的成员值
- 右键删除实例 → 渐隐 + 检查内存状态

#### 方向 D：自由探索关卡

每 2-3 个引导关卡后插入一个开放关卡。
- 不设空白，提供完整代码编辑器
- 任务从「填空」变为指令式：「让世界同时出现一只苦力怕和一只铁傀儡，其中苦力怕的 health 是铁傀儡的两倍」
- 用户可以写任意代码，Canvas 实时反馈

#### 方向 E：合成台隐喻

MC 最核心的交互模型：3×3 合成格。
- 把 class 定义的各个元素（`class`、类名、成员类型、成员名）做成可拖拽的材料
- 拖进合成格 → 拼出合法类定义 → Canvas 上合成出一个实例
- 不再是敲代码填空，而是「操作配方 → 出成品」

### 2.3 方向评估矩阵

| 方向 | 趣味性 | 教学效果 | 实现成本 | 与 MC 融合度 |
|---|---|---|---|---|
| A. 即时沙盒 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 中 | ⭐⭐⭐ |
| B. 生存机制 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 低 | ⭐⭐⭐⭐⭐ |
| C. 交互 Canvas | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 高 | ⭐⭐⭐ |
| D. 自由探索 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 中 | ⭐⭐ |
| E. 合成台 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 高 | ⭐⭐⭐⭐⭐ |

## 三、讨论议题

1. **选择题是否保留？** 如果保留，是否需要生存机制让它不那么像考试？
2. **Canvas 的优先级？** 是先把静态 Canvas 做精致（卡片样式、布局），还是先做交互式 Canvas？
3. **即时沙盒 vs 填空题？** 填空题好实现但交互密度低，即时沙盒交互密度高但需要改写 Engine。两者能否共存（引导关卡用填空，自由关卡用即时沙盒）？
4. **MC 机制的深度？** 旧 demo 的生存模式/心形/难度/连击花了 ~200 行代码，要不要现在引入？
5. **下一步先做哪个？**

## 三、讨论决策（2026-05-14 组内决议）

### 3.1 选择题 — 坚决砍掉

选择题是破坏「世界建造感」的罪魁祸首。从关卡 JSON 中移除 `questions` 字段，NPC 对话直接过渡到填空代码。

### 3.2 Canvas — 静态做精，交互延后

方向 C（交互式 Canvas）和方向 E（合成台）是工程黑洞，不碰。把精力投入到：
- 接入已写好的飞入/闪烁/渐隐动画
- 接入 MC 经典音效（放置方块、经验球、升级、受伤）
- 粒子特效（卡片出现时播放「噔」音效）

### 3.3 即时沙盒 — 方向 A 的灵魂保留，但要「取巧」

不做逐字符解析。做「伪即时沙盒」：用户敲完一行的瞬间（Enter 或失去焦点），触发预设的 Canvas 渲染。底层按行触发断点，感觉是即时的。

### 3.4 MC 生存机制 — 白嫖旧代码，满上

旧 demo 的 200 行心形生命值、难度模式、连击系统直接接入。执行力：
- 执行失败 → 扣血 + 屏幕红闪 + 受伤音效
- 执行成功 → 连击 +1 → XP 音效，连击 3 次 → 升级音效
- 心耗光 → 挑战失败动画

## 四、Sprint 冲刺计划

### Sprint 1：合体与反馈（方向 B + 视觉接入）

- 接入生存机制（心形生命值、连击、扣血）
- 接入 CanvasAnimator 飞入/渐隐动画
- 接入 MC 音效（受伤、XP、升级、死亡）
- 砍掉选择题

### Sprint 2：重塑核心体验（方向 A 青春版）

- 把填空题「点击执行」→ 改为「每行输入完成自动触发渲染」
- 实现 LearnGitBranching 的初步感觉：左边敲代码，右边立刻出卡片

### Sprint 3：降维打击（方向 D 自由探索）—— 如时间允许

- 做一个无空白的自由编辑器页面
- 只设目标（如「实例化三个不同血量的对象」）
- Engine 跑最后状态核对
