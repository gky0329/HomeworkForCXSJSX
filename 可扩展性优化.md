# 可扩展性优化 — 插件化架构设计

> 版本 v1.0 | 2026-05-13
>
> 目标：不修改任何 Python 代码的情况下，仅通过新增 JSON 文件即可发布新关卡和新判定规则。

---

## 一、核心原则

### 1.1 数据驱动设计 (Data-Driven Design)

**关卡逻辑 = 纯配置文件。** Python 代码只做「无情的播放器和判题机器」。

```
新人贡献流程:
  复制 _template.json → 修改字段 → 放进 data/levels/ → 完成
  (0 行 Python 代码)
```

### 1.2 YAGNI 原则 (You Aren't Gonna Need It)

| ✅ MVP 必须做 | ❌ MVP 不碰 |
|---|---|
| 关卡 JSON 全配置 | 可视化关卡编辑器 |
| Goal Checker 注册表 | 完整 C++ AST 解析器 |
| Parser 黑盒封装 | 多作用域嵌套解析 |
| Canvas 工厂接口预留 | PointerArrow / MemoryBlock 实现 |
| `_template.json` 贡献模板 | 在线关卡市场 / Web 编辑器 |

**为扩展性留好「插槽（接口）」，但不去实现未来可能需要的复杂功能。**

### 1.3 开闭原则 (Open-Closed Principle)

> 对扩展开放，对修改关闭。

- **新增判定类型**：在 `CHECKER_REGISTRY` 字典加一行 → 不改老代码
- **新增可视化元素**：在 `ITEM_FACTORY` 字典加一行 → 不改老代码
- **新增关卡**：在 `data/levels/` 加一个 JSON → 不改任何 Python 文件

## 二、插件化架构设计

### 2.1 Goal Checker — 注册表模式

**现状 (反模式)**：`challenge_loader.py` 内嵌 `if-elif` 链。

```python
# ❌ 反模式：未来每加一种判定，就要改这个函数
def _run_check(check, model):
    if ct == "class_exists": ...
    elif ct == "instance_exists": ...
    elif ct == "member_value_equals": ...
    elif ct == "instance_count": ...
    # 未来: elif ct == "inherits_from": ...
```

**设计 (注册表模式)**：

```python
# ✅ 正确：判定函数注册表
def _check_class_exists(check: dict, model: OopModel) -> tuple[bool, str]:
    name = check["name"]
    ok = name in model.classes
    return ok, f"类 {name} {'存在' if ok else '不存在'}"

def _check_instance_exists(check: dict, model: OopModel) -> tuple[bool, str]:
    inst = model.get_instance(check["var_name"])
    ok = inst is not None and inst.cls_name == check["class_name"]
    return ok, f"实例 {check['var_name']} {'存在' if ok else '不存在或类型不匹配'}"

def _check_member_value_equals(check: dict, model: OopModel) -> tuple[bool, str]:
    inst = model.get_instance(check["var_name"])
    if inst is None:
        return False, f"实例 {check['var_name']} 不存在"
    val = inst.values.get(check["member"])
    expected = check["value"]
    ok = val == expected
    return ok, f"{check['var_name']}.{check['member']} == {val}, 期望 {expected}"

# ⭐ 注册表：未来加新判定只需加一行
CHECKER_REGISTRY: dict[str, Callable] = {
    "class_exists":           _check_class_exists,
    "class_has_member":       _check_class_has_member,
    "instance_exists":        _check_instance_exists,
    "instance_count":         _check_instance_count,
    "member_value_equals":    _check_member_value_equals,
    "no_errors":              _check_no_errors,
}

def run_check(check: dict, model: OopModel) -> tuple[bool, str]:
    check_type = check["type"]
    if check_type not in CHECKER_REGISTRY:
        return False, f"未知检查类型: {check_type}"
    return CHECKER_REGISTRY[check_type](check, model)
```

**未来扩展示例**：只需在注册表加一行，不改任何已有代码。

```python
# Phase 2 新增继承检查
CHECKER_REGISTRY["inherits_from"] = _check_inherits_from
CHECKER_REGISTRY["constructor_called"] = _check_constructor_called
```

### 2.2 Parser — 黑盒封装

**现状 (反模式)**：UI 和测试直接调用 `step_executor.execute_line()`。

```python
# ❌ 反模式：多处直接调用内部函数
from app.core.step_executor import execute_line
diff = execute_line(code, line_number, model)
```

**设计 (黑盒接口)**：

```python
# ✅ app/core/engine.py — 唯一的对外接口
class Engine:
    """OOP 代码执行引擎。
    
    替代方案说明：当前内部使用正则匹配。如果未来替换为 tree-sitter
    或 Clang AST，只需修改本文件内部的 execute_line 实现，
    UI 层和关卡系统完全不需要感知变化。
    """

    def __init__(self, model: OopModel | None = None):
        self.model = model or OopModel()

    def execute(self, code_lines: list[str]) -> list[StateDiff]:
        """执行一个完整的代码块，返回每行的 StateDiff 列表。"""
        diffs = []
        for i, line in enumerate(code_lines):
            diff = execute_line(line, i, self.model)
            diffs.append(diff)
        return diffs

    def execute_single(self, code: str, line_number: int) -> StateDiff:
        """执行单行代码。"""
        return execute_line(code, line_number, self.model)

    def reset(self) -> None:
        """重置引擎状态。"""
        self.model.reset()

    def snapshot(self) -> OopModel:
        """获取当前状态的深拷贝。"""
        return self.model.snapshot()
```

**UI 层调用约定**：

```python
# UI 层只调 engine.execute()，不碰 execute_line
engine = Engine()
diffs = engine.execute(code_lines)
canvas.apply(diffs)
```

### 2.3 Canvas — 工厂模式（预留设计）

**设计 (工厂接口)**：

```python
# ✅ app/ui/canvas/item_factory.py — 按 diff 类型分发 QGraphicsItem
from app.core.state_diff import StateDiff
from app.ui.canvas.class_item import ClassItem
from app.ui.canvas.instance_item import InstanceItem

def create_item(diff: StateDiff) -> list[QGraphicsItem]:
    """根据 StateDiff 内容生成对应的画布图形项。"""
    items = []

    for cd in diff.added_classes:
        items.append(ClassItem(cd))

    for inst in diff.added_instances:
        items.append(InstanceItem(inst))

    for var, prop, val in diff.updated_instances:
        items.extend(InstanceItem.update_highlight(var, prop, val))

    for name in diff.removed_instances:
        items.extend(InstanceItem.fade_out(name))

    return items
```

**未来扩展**：新增 PointerArrow、MemoryBlock 时，只需在此函数加分支，不改其他文件。

### 3.3 必填字段 vs 可选字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | 唯一标识，如 `oop_001` |
| `title` | string | ✅ | 关卡标题 |
| `type` | string | ✅ | `challenge` / `quiz` / `sandbox` |
| `unit` | int | ✅ | 所属学习单元编号 |
| `order` | int | ✅ | 单元内的顺序 |
| `prerequisites` | list | ✅ | 前置关卡 id 列表 |
| `initial_code` | list | ✅ | 预填代码行 |
| `blanks` | list | ✅ | 挖空定义 |
| `goal.checks` | list | ✅ | 过关判定条件 |
| `theory` | string | 推荐 | 教学文本 |
| `key_points` | list | 推荐 | 关键知识点 |
| `hints` | list | 推荐 | 3 条渐进提示 |
| `success_message` | string | 推荐 | 通关文案 |
| `npc_dialogue` | list | 可选 | NPC 对话（无则跳过 lesson_intro） |
| `code_line_notes` | list | 可选 | 每行代码注释 |
| `failure_guidance` | string | 可选 | 失败时的补充说明 |
| `estimated_time_min` | int | 可选 | 预计耗时 |

## 四、贡献者指南 (Contributing Guide)

### 4.1 0 行代码发布新关卡

```
1. 复制 data/levels/_template.json
2. 重命名为 oop_NNN.json
3. 修改以下字段:
   - id:            唯一 ID
   - title:         关卡标题
   - unit/order:    关卡位置
   - theory:        教学文本
   - initial_code:  预填代码（____ 为挖空处）
   - blanks:        每个 blank 的 validator 和 placeholder
   - goal.checks:   通关条件（从 CHECKER_REGISTRY 中选择）
   - hints:         3 条渐进提示
4. 在 data/levels/index.json 中注册
5. 提交 PR
```

### 4.2 如何新增一种判定类型

如果现有 Checker 不够用（如需要「检查继承链」）：

```
1. 在 CHECKER_REGISTRY 中注册新函数（1 行代码）
2. 在关卡 JSON 的 goal.checks 中引用新 type
3. 更新本文档的 CHECKER_REGISTRY 列表
```

### 4.3 Validator 类型速查

| Validator | 允许输入 | 示例 |
|---|---|---|
| `cpp_keyword` | `class`, `struct`, `public`, `private` | class |
| `cpp_access` | `public`, `private`, `protected` | public |
| `identifier` | `[A-Za-z_][A-Za-z0-9_]*` | health |
| `number` | `[0-9]+` | 20 |
| `string_literal` | `"任意文本"` | "村庄安全了!" |
