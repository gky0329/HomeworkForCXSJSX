# C++ 内存实验室 — 关卡设计准则

> 版本 v1.0 | 2026-05-10
>
> 每条准则必须可验证。各级关系卡在发布前需通过本清单逐项自检。

---

## 一、核心原则

### 1.1 四问原则（每关必须回答）

用户打开关卡时，必须在 **30 秒内** 从界面上直接获得以下四个问题的答案：

| 问题 | 界面对应区域 | 负面案例 |
|---|---|---|
| **我在学什么？** | 学习目标列表 | 「声明你的第一根指针」——标题模糊，未说明学的是 `int*` 语法 |
| **为什么要学这个？** | 理论讲解第一段 | 没有动机铺垫，直接给语法 |
| **我要做什么？** | 任务指令框 | `__BLANK__` 孤立出现，用户不知道要填什么 |
| **做对了吗？** | Canvas 视觉反馈 + 通关提示 | 只弹 QMessageBox，用户不知道 Canvas 上的箭头代表什么 |

### 1.2 信息密度原则

面向大学生，每关教学文本 150-400 字：

- **太少**（< 100 字）：用户缺少认知框架，变成瞎猜填空
- **太多**（> 500 字）：认知负荷过高，变成看文档而非交互学习
- **正确区间**：150-400 字，结构化为「类比引入(50字) → 概念定义(100字) → 内存行为(100字) → 本关任务(50字)」

### 1.3 单一概念原则

每关只引入 **1 个新语法元素**：

| 关 | 新元素 | 已学 |
|---|---|---|
| ptr_001 | `int*` 指针声明 | `int a = 42;` 变量声明 |
| ptr_002 | `&` 取地址 | `int*` |
| ptr_003 | `*` 解引用 | `int*`, `&` |
| ptr_004 | `new` 堆分配 | `int*`, `&`, `*` |
| ptr_005 | `delete` 释放 | `new` |

> 反例：一关同时教 `&` 和 `*` —— 用户无法区分自己的错误是因为不理解 `&` 还是 `*`。

## 二、教学文本编写规范

### 2.1 theory 字段结构模板

```
[类比引入] 用生活场景类比（排盒、门牌号、快递柜、停车位）
            → 50字以内

[概念定义] 精确定义本关的语法元素，一句话
            → 50字以内

[内存行为] 描述这行代码在内存中做了什么
            → 100字以内
            → 关键：描述用户可以「看到」什么（Canvas 上的变化）

[关键规则] 1-2 条容易被忽略的约束
            → 50字以内
```

### 2.2 类比库（可复用）

| 概念 | 类比 | 适用关卡 |
|---|---|---|
| 变量 | 一个带标签的盒子，里面可以放东西 | ptr_001 |
| 内存地址 | 盒子在仓库里的货架编号 | ptr_002 |
| 指针 | 一张写了货架编号的纸条 | ptr_001 |
| `&` 取地址 | 查看盒子的货架编号 | ptr_002 |
| `*` 解引用 | 按照纸条上的编号找到那个盒子，打开它 | ptr_003 |
| 栈 | 收银台旁边的临时存放区，离开柜台就清空 | ptr_004 |
| 堆 | 仓库的长期存储区，不主动清理就一直占着 | ptr_004 |
| `new` | 在仓库里申请一个新货架 | ptr_004 |
| `delete` | 把货架归还，让别人可以用 | ptr_005 |
| 内存泄漏 | 货架被占了但没人知道在哪，永远拿不回来 | ptr_005 |
| 悬挂指针 | 纸条上的货架编号指向了一个已经归还的空位 | ptr_005 |
| `int*` | 「这张纸条上写的号码，对应的是一个 int 类型的盒子」 | ptr_001 |

### 2.3 禁用表述

| ❌ 禁用 | ✅ 替换为 | 原因 |
|---|---|---|
| 「大家应该都知道」 | 直接给出定义 | 预设知识 |
| 「显然」「不难发现」 | 去掉或换成「可以看到」 | 对初学者不友好 |
| 「指针就是地址」 | 「指针是一个变量，它**存储**另一个变量的地址」 | 概念不精确 |
| 「在内存中分配」 | 「在栈/堆上创建了一块空间」 | 不指定区域 |
| 不提 Canvas 变化 | 「按下下一步后，Canvas 的栈区会出现...」 | 缺少视觉指引 |

## 三、UI 引导设计

### 3.1 信息层级（从上到下）

```
┌─────────────────────────────┐
│ 1. 关卡标题 + 学习目标        │  ← 最小字体 / 次要颜色
│    「学完本关你将能够...」    │
├─────────────────────────────┤
│ 2. 理论讲解                  │  ← 主阅读区，正常字体
│    （类比 → 定义 → 行为）     │
├─────────────────────────────┤
│ 3. 关键知识点（高亮卡片）      │  ← 加粗 / 背景色块
│    - int* 声明指针变量       │
│    - & 获取变量地址          │
├─────────────────────────────┤
│ 4. 任务指令（醒目区域）        │  ← 明亮颜色 / 左边框
│    「请在第3行填写代码...」   │
├─────────────────────────────┤
│ 5. 代码面板                  │  ← 深色背景 + 行号
│    int a = 42;  // 注释      │
│    ┌──────────┐             │
│    │__BLANK__ │             │
│    └──────────┘             │
├─────────────────────────────┤
│ 6. 提示按钮（可折叠）         │  ← 「需要提示？」点击展开
│    提示1: ...                │
│    提示2: ...                │
│    提示3: ...                │
├─────────────────────────────┤
│ 7. 控制栏                    │
│    [上一步] [▶ 下一步] [重置] │
└─────────────────────────────┘
```

### 3.2 渐进式提示（3 级）

```
提示 1: 方向性提示 —— 引导用户思考，不透露答案
        例：「你需要一个新变量，它的类型和 int 不同」
        
提示 2: 工具性提示 —— 提示用到的语法元素
        例：「用 int* 声明一个指针，用 & 获取变量地址」
        
提示 3: 模板性提示 —— 给出语句模板
        例：「int* p = &a;」
```

用户每次点击「需要提示？」按钮（或按 H 键），展开下一级提示。避免一次性曝光。

### 3.3 Canvas 视觉反馈（每步之后自动触发）

| 引擎事件 | Canvas 表现 | 教学暗示 |
|---|---|---|
| 变量声明 | 栈帧中出现新行，带类型标注 | 「变量已经在内存中创建了」 |
| 指针声明 | 栈帧中出现 `int*` 类型行，灰色虚线 | 「指针已声明，但还没指向任何东西」 |
| `p = &a` | 蓝色实线箭头从 p 指向 a，箭头生长动画 | 「p 现在存储了 a 的地址」 |
| `*p = 100` | a 的值从 42 动画过渡到 100，高亮闪烁 | 「通过指针间接修改了 a」 |
| `new int(5)` | 橙色方块飞入堆区，箭头指向它 | 「堆上分配了新内存」 |
| `delete p` | 橙色方块碎裂消失，箭头变红虚线 | 「内存已归还，但指针还在（悬挂）！」|
| 内存泄漏 | 堆块加红色虚线边框 + ⚠ 图标 | 「这块内存没人管了」 |

### 3.4 通关 / 失败反馈

**通关**：
- Canvas 正确状态保持 1 秒
- 弹出半透明通关横幅（非阻塞）：`✓ 通关！[success_message]`
- 自动高亮下一关按钮

**失败**：
- 对比式反馈：「你写的 `xxx` 让 p 指向了堆区，但本关要求指向栈上的 a」
- 错误位置高亮：对应的代码行标红
- Canvas 异常状态闪烁（悬挂指针标红、泄漏块标橙）

## 四、代码注释规范（code_line_notes）

每行预填代码必须有注释，格式为行内 `//` 风格。注释不解释语法，解释**这行代码在内存模型中触发了什么**。

```
int main() {          // 进入 main 函数的栈帧
    int a = 42;       // 在 main 栈帧中创建变量 a，写入 42
    int* p = &a;      // 创建指针 p，存储 a 的地址 ← 这是本关新概念
}
```

**编写规则**：
1. 每个非空行都要有注释
2. 本关新学的那一行，注释用 `←` 箭头特别标注
3. 已学过的概念，注释可以更简短
4. 注释不写「为什么要这样做」（那是 theory 的职责），只写「触发了什么」

## 五、关卡自检清单

发布前逐项确认：

### 结构
- [ ] 学习目标 ≤ 5 条
- [ ] 理论文本 150-400 字
- [ ] 只引入 1 个新语法元素
- [ ] 任务指令 1-2 句话
- [ ] 3 条渐进提示
- [ ] 通关文案解释了 Canvas 上的变化

### 教学
- [ ] theory 第一段包含类比
- [ ] 描述了「Canvas 上会看到什么」
- [ ] 关键知识点有高亮标注（至少 2 条）
- [ ] 代码注释覆盖每一行
- [ ] 本关新概念那一行有 `←` 特别标注

### UX
- [ ] 用户不看提示也能从 theory + task 推断出任务
- [ ] 看完提示 1 用户能知道方向
- [ ] 看完提示 3 用户能写出正确代码
- [ ] Canvas 变化在 success_message 中有解释
- [ ] 失败反馈不只是「未通过」，而是说明了为什么

### 技术
- [ ] goal checks 覆盖正确解法的所有关键状态
- [ ] 常见错误的 goal check 反馈有意义（不报「未知错误」）
- [ ] 如果本关有 `initial_code` 中的代码，它们能被 StepExecutor 正确解析
- [ ] 用户输入的典型错误（例如用 new 代替 &）能被检测到并给出明确指引

## 六、反例警示

### 反例 A：信息过少
```json
{
  "title": "声明指针",
  "concept": "int*",
  "initial_code": ["int a = 42;", "__BLANK__"],
  "hints": ["提示：int* p = &a;"]
}
```
用户不知道 `int*` 和 `int` 有什么区别，不知道为什么需要 `&`。

### 反例 B：多个新概念
```json
{
  "title": "指针和动态内存",
  "task": "声明指针，取地址，new 堆分配",
  "initial_code": ["int a = 42;", "__BLANK__", "__BLANK__"]
}
```
用户同时面对 `int*`、`&`、`new` 三个陌生符号，失败时不知道自己错在哪里。

### 反例 C：类比错误
```
theory: "指针就像一个指向内存的箭头"
```
箭头是 Canvas 上的视觉表现，不是指针本身。正确类比：「指针是一个存地址的变量，就像纸条上写了一个货架编号。Canvas 上画出的箭头是帮助你看清『纸条指向哪个货架』。」

## 七、UX 想象力边界

以下交互暂不在 MVP 实现，但作为设计储备，在后续迭代中考虑：

| 交互 | 描述 | 优先级 |
|---|---|---|
| **动画倒放** | 撤销操作时，动画反向播放（箭头缩回、堆块飞回代码行） | 中 |
| **自由探索模式** | 通关后解锁，用户可自由输入任意 C++ 代码观察内存变化 | 高 |
| **错误预测** | 在用户提交前，Engine 预执行并提前标出潜在问题（对比 LGB 的 `git status` 预览） | 低 |
| **成就系统** | 「零提示通关」「一次通过」「发现 3 种不同解法」 | 低 |
| **多人对比** | 同一关不同用户的解法并排展示 | 低 |
