# C++ OOP 教学工具 — 关卡设计准则

> 版本 v2.1 | 2026-05-13
>
> 每条准则必须可验证。关卡发布前需通过本清单逐项自检。

---

## 一、核心原则

### 1.1 四问原则（每关必须回答）

用户打开关卡时，必须在 **30 秒内** 从界面上直接获得以下四个问题的答案：

| 问题 | 界面对应区域 | 负面案例 |
|---|---|---|
| **我在学什么？** | 学习目标列表 | 「定义你的第一个类」——标题模糊，未说明学的是 `class` 关键字 |
| **为什么要学这个？** | 理论讲解第一段 | 没有动机铺垫，直接给语法 |
| **我要做什么？** | 任务指令框 | `__BLANK__` 孤立出现，用户不知道要填 `class` 还是 `struct` |
| **做对了吗？** | Canvas 视觉反馈 + 通关提示 | 只弹 QMessageBox，用户不知道 Canvas 上的类卡片和实例卡片代表什么 |

### 1.2 信息密度原则

面向大学生，每关教学文本 150-400 字：

- **太少**（< 100 字）：用户缺少认知框架，变成瞎猜填空
- **太多**（> 500 字）：认知负荷过高，变成看文档而非交互学习
- **正确区间**：150-400 字，结构化为「类比引入(50字) → 概念定义(100字) → OOP 行为(100字) → 本关任务(50字)」

### 1.3 单一概念原则

每关只引入 **1 个新语法元素**：

| 关 | 新元素 | 已学 |
|---|---|---|
| oop_001 | `class` 类声明 + 成员变量 | — |
| oop_002 | 多类定义 + 多实例 | `class`、成员变量 |
| oop_003 | `public:` / `private:` 访问控制 | 类定义、实例化 |
| oop_004 | 构造函数 | 类定义、访问控制 |
| oop_005 | 继承 | 类定义、构造、访问控制 |

> 反例：一关同时教 `class` 和 `public/private` —— 用户无法区分自己的错误是因为不理解类定义语法还是不熟悉访问控制。

### 1.4 视觉锚点原则（防语法「全家桶」干扰）

C++ 类定义的语法天然捆绑：即使只想教 `class` 关键字，也避不开 `{}` 和 `;`。为保护「单一概念原则」，采用以下策略：

**将尚未教学的语法元素预填固化**，不作为填空项：

| 元素 | 策略 | 原因 |
|---|---|---|
| `{}` 大括号 | **预填**，显示在代码面板中 | 教 `class` 时不希望用户因 `{}` 位置错误而卡关 |
| `;` 分号 | **预填**，显示在代码面板中 | 这是语法琐碎，不是概念理解 |
| `int` 类型关键字 | **前几关预填**，后续逐步放开 | 先让用户理解类和对象的逻辑关系，再扩展类型选择 |
| 缩进空白 | **预填** | 不增加无关认知负荷 |

**唯一填空区域**：严格限定为**本关新引入的那个语法元素**。例如 oop_001 的第 1 个 blank 只能填 `class`，不要同时让用户填 `Player`（类名）、`public:`（访问控制）或成员名。

**代码面板呈现规范**：用户不可编辑的行用灰色锁图标 🔒 标记，暗示「这部分是固定语法，以后会学到」。这既保持了完整代码的可读性，又防止用户纠结于不相关的语法细节。

## 二、教学文本编写规范

### 2.1 theory 字段结构模板

```
[类比引入] 用生活场景类比（蓝图与房屋、食谱与菜肴、模具与成品）
            → 50字以内

[概念定义] 精确定义本关的语法元素，一句话
            → 50字以内

[OOP 行为] 描述这行代码在 OOP 模型中触发了什么
            → 100字以内
            → 关键：描述用户可以「看到」什么（Canvas 上的变化）

[关键规则] 1-2 条容易被忽略的约束
            → 50字以内
```

### 2.2 类比库（可复用）

| 概念 | 类比 | 适用关卡 |
|---|---|---|
| 类 (class) | 一张蓝图，描述一类事物的共同特征 | oop_001 |
| 对象 (object) | 根据蓝图建造出来的具体房屋 | oop_001 |
| 成员变量 | 蓝图上标注的属性，如「这栋房子有 3 个房间」 | oop_001 |
| 实例化 | 拿着蓝图去施工，造出一栋真实的房子 | oop_001 |
| `public:` | 房子的客厅——外面的人可以进来 | oop_003 |
| `private:` | 房子的卧室——只有内部成员能进 | oop_003 |
| 构造函数 | 房子交付前的初始化工作：通水电、刷墙 | oop_004 |
| 继承 | 在已有蓝图基础上加一层，新蓝图继承旧蓝图的所有设计 | oop_005 |
| 成员函数 | 房子能做的事情：开门、亮灯、供暖 | 后续 |

### 2.3 禁用表述

| ❌ 禁用 | ✅ 替换为 | 原因 |
|---|---|---|
| 「大家应该都知道」 | 直接给出定义 | 预设知识 |
| 「显然」「不难发现」 | 去掉或换成「可以看到」 | 对初学者不友好 |
| 「class 就是类型」 | 「class 是一个自定义的类型模板，描述了该类对象有哪些数据和行为」 | 概念不精确 |
| 「创建对象」 | 「根据类定义实例化一个对象」 | 不说明来源 |
| 不提 Canvas 变化 | 「按下下一步后，Canvas 的 Class 区会出现...」 | 缺少视觉指引 |

## 三、UI 引导设计

### 3.1 信息层级（从上到下）

```
┌─────────────────────────────┐
│ 1. 关卡标题 + 学习目标        │  ← 最小字体 / 次要颜色
│    「学完本关你将能够...」    │
├─────────────────────────────┤
│ 2. 理论讲解                  │  ← 主阅读区，正常字体
│    （类比 → 定义 → 行为）     │
├─────────────────────────────┤
│ 3. 关键知识点（高亮卡片）      │  ← 加粗 / 背景色块
│    - class 声明类             │
│    - . 运算符访问成员         │
├─────────────────────────────┤
│ 4. 任务指令（醒目区域）        │  ← 明亮颜色 / 左边框
│    「请在第1行填写类关键字...」│
├─────────────────────────────┤
│ 5. 代码面板                  │  ← 深色背景 + 行号
│    ____ Player {             │
│    public:                   │
│        int health;           │
│    };                        │
│    Player p;                 │
├─────────────────────────────┤
│ 6. 提示按钮（可折叠）         │  ← 「需要提示？」点击展开
│    提示1: ...                │
│    提示2: ...                │
│    提示3: ...                │
├─────────────────────────────┤
│ 7. 控制栏                    │
│    [上一步] [▶ 下一步] [重置] │
└─────────────────────────────┘
```

### 3.2 渐进式提示（3 级）

```
提示 1: 方向性提示 —— 引导用户思考，不透露答案
        例：「你需要一个自定义类型，它可以把数据和属性打包在一起」
        
提示 2: 工具性提示 —— 提示用到的语法元素
        例：「用 class 关键字定义类，类体用 {} 括起来」
        
提示 3: 模板性提示 —— 给出语句模板
        例：「class Player {\npublic:\n    int health;\n};」
```

用户每次点击「需要提示？」按钮（或按 H 键），展开下一级提示。避免一次性曝光。

### 3.3 Canvas 视觉反馈（每步之后自动触发）

| 引擎事件 | Canvas 表现 | 教学暗示 |
|---|---|---|
| 类定义 | Class 区出现绿色卡片，含类名 + 成员列表 | 「类已经定义好了，是一张蓝图」 |
| `public:` | 成员旁的锁图标从 🔒 变为 🔓 | 「现在外部可以访问这些成员了」 |
| 对象实例化 | Instance 区飞出蓝色卡片，从类卡片位置分裂出来 | 「对象是根据蓝图创建的具体实例」 |
| 成员赋值 | 实例卡片上对应的值闪烁高亮，数字过渡动画 | 「修改了对象的一个属性」 |
| 作用域退出 `}` | 所有实例卡片渐隐消失 | 「离开作用域，局部对象被销毁」 |
| 访问越权 | 目标成员标红 + ⚠ 图标 | 「这个成员是 private 的，外部不能访问」 |

### 3.4 通关 / 失败反馈

**通关**：
- Canvas 正确状态保持 1 秒
- 弹出半透明通关横幅（非阻塞）：`✓ 通关！[success_message]`
- 自动高亮下一关按钮

**失败 —— 严禁抛出原始错误信息**：
- ⛔ **禁止**：`error: expected ';' after class definition`
- ✅ **必须**：`「蓝图的结尾需要分号(;)作为句号，在 ; 处填上它」`
- ⛔ **禁止**：`SyntaxError at line 3`
- ✅ **必须**：`「第 3 行的 class 关键字拼写错误，请检查」`

**失败反馈三要素**：
1. **对比式诊断**：「你写的是 `struct`，但本关要求使用 `class` 关键字。两者默认访问控制不同。」
2. **位置标红**：对应的代码行高亮为红色
3. **Canvas 异常状态**：未定义类标红虚线边框、访问越权标黄色警告图标、实例创建失败不显示任何卡片

### 3.5 Canvas 空间管理（防视觉混乱）

在多类多实例关卡中，Canvas 可能变得拥挤。制定以下布局策略：

| 场景 | 策略 |
|---|---|
| Class 区 → 超过 4 个类 | 自动缩小卡片为紧凑模式（只显示类名，成员折叠） |
| Instance 区 → 超过 6 个实例 | 多余实例折叠为「+N 个实例」标签，点击展开 |
| 某类 ≥ 5 个成员 | 成员列表启用纵向滚动 |
| 窗口最小宽度 < 800px | Class 区和 Instance 区上下堆叠排列 |

## 四、代码注释规范（code_line_notes）

每行预填代码必须有注释，格式为行内 `//` 风格。注释不解释语法，解释**这行代码在 OOP 模型中触发了什么**。

```
int main() {          // 进入 main 作用域
    class Player {    // 定义一个名为 Player 的类 ← 本关新概念
    public:           // 以下成员可被外部访问
        int health;   // Player 有一个 int 类型的 health 成员
    };
    Player p;         // 根据 Player 蓝图创建一个实例 p
    p.health = 20;    // 设置 p 的 health 为 20
}
```

**编写规则**：
1. 每个非空行都要有注释
2. 本关新学的那一行，注释用 `←` 箭头特别标注
3. 已学过的概念，注释可以更简短
4. 注释不写「为什么要这样做」（那是 theory 的职责），只写「触发了什么」

## 五、关卡自检清单

发布前逐项确认：

### 结构
- [ ] 学习目标 ≤ 5 条
- [ ] 理论文本 150-400 字
- [ ] 只引入 1 个新语法元素
- [ ] 任务指令 1-2 句话
- [ ] 3 条渐进提示
- [ ] 通关文案解释了 Canvas 上的变化

### 教学
- [ ] theory 第一段包含类比（蓝图、食谱等）
- [ ] 描述了「Canvas 上会看到什么」
- [ ] 关键知识点有高亮标注（至少 2 条）
- [ ] 代码注释覆盖每一行
- [ ] 本关新概念那一行有 `←` 特别标注

### UX
- [ ] 用户不看提示也能从 theory + task 推断出任务
- [ ] 看完提示 1 用户能知道方向
- [ ] 看完提示 3 用户能写出正确代码
- [ ] Canvas 变化在 success_message 中有解释
- [ ] 失败反馈不只是「未通过」，而是说明了为什么

### 技术
- [ ] goal checks 覆盖正确解法的所有关键状态（class_exists, instance_exists, member_value_equals 等）
- [ ] 常见错误的 goal check 反馈有意义（不报「未知错误」）
- [ ] 如果本关有 `initial_code` 中的代码，它们能被 StepExecutor 正确解析
- [ ] 用户输入的典型错误（例如用 struct 代替 class）能被检测到并给出明确指引

## 六、反例警示

### 反例 A：信息过少
```json
{
  "title": "定义一个类",
  "concept": "class",
  "initial_code": ["____ Player {", "    int health;", "};"],
  "hints": ["提示：填 class"]
}
```
用户不知道 `class` 和 `struct` 有什么区别，不知道为什么需要 `{}`。

### 反例 B：多个新概念
```json
{
  "title": "类和访问控制",
  "task": "定义类，声明 public 和 private 成员",
  "initial_code": ["____ Player {", "____:", "    int health;", "};"]
}
```
用户同时面对 `class`、`public:`、`private:` 三个陌生符号，失败时不知道自己错在哪里。

### 反例 C：类比错误
```
theory: "class 就像一个对象"
```
class 是类型模板，不是对象本身。正确类比：「class 是建筑蓝图，定义了这栋房子有哪些房间。对象是根据蓝图实际建造出来的那栋房子。」

## 七、UX 想象力边界

以下交互暂不在 MVP 实现，但作为设计储备，在后续迭代中考虑：

| 交互 | 描述 | 优先级 |
|---|---|---|
| **自由探索关卡** | 每 3-5 个引导关卡后插入一个开放关卡，不设 `__BLANK__`，提供完整编辑器。任务从填空转为指令式，如「设计一个包含两个私有成员的矩形类」。弥补填空模式对设计能力培养的不足。 | **高** |
| **动画倒放** | 撤销操作时，动画反向播放（实例缩回类卡片、值回退） | 中 |
| **错误预测** | 在用户提交前，Engine 预执行并提前标出潜在问题 | 低 |
| **成就系统** | 「零提示通关」「一次通过」「发现 3 种不同写法」 | 低 |
| **继承树可视化** | 展示父类-子类的层级关系图 | 中 |

### 7.1 填空模式的局限性（坦诚说明）

本工具当前以**填空模式**为核心交互。填空模式擅长教授**语法和单点概念**，但存在以下天花板：

| 能做 | 不能做（或做不好） |
|---|---|
| 让用户识别 `class` 是定义类的关键字 | 让用户思考「为什么这个场景需要定义一个新类」 |
| 让用户正确写出成员赋值 `p.health = 20` | 让用户自主设计类的成员结构 |
| 验证用户对 `public/private` 的理解 | 培养面向对象的设计思维和封装直觉 |

**应对策略**：
1. 在引导关卡中通过 **theory 文本 + NPC 对话** 解释设计动机（为什么定义这个类）
2. 在 **自由探索关卡**（见上表）中提供开放式设计任务
3. 后续 Phase 引入**半开放填空**：用户不仅填关键字，还可选择成员类型和名称
